═══════════════════════════════════════════════
WHAT YOU MUST CHECK
═══════════════════════════════════════════════

1. NAMING
   - Vague or misleading variable names (x, data, temp, foo, flag)
   - Functions named after implementation not intent (process_it vs validate_user_email)
   - Boolean variables not named as predicates (is_, has_, can_, should_)
   - Magic numbers and strings without named constants
   - Inconsistent naming conventions across the codebase

2. FUNCTIONS & METHODS
   - Functions doing more than one thing (violates Single Responsibility)
   - Functions longer than ~20 lines (should be broken down)
   - Too many parameters (>3-4 suggests a missing abstraction)
   - Functions with side effects that are not obvious from the name
   - Boolean flag parameters that change function behaviour (use two functions instead)
   - Deep nesting (>3 levels — use early returns / guard clauses)

3. SOLID PRINCIPLES
   - S: Single Responsibility — class/function doing too many things
   - O: Open/Closed — code requiring modification to extend behaviour
   - L: Liskov Substitution — subclass breaking parent contract
   - I: Interface Segregation — fat interfaces forcing unwanted dependencies
   - D: Dependency Inversion — high-level modules depending on low-level details

4. DRY (DON'T REPEAT YOURSELF)
   - Duplicated logic that should be a shared function
   - Copy-pasted blocks with minor variations
   - Same validation logic in multiple places
   - Constants defined multiple times

5. COMMENTS & DOCUMENTATION
   - Missing docstrings on public functions/classes
   - Comments explaining WHAT instead of WHY (the code should explain what)
   - Outdated or misleading comments
   - Commented-out dead code that should be deleted
   - Missing type hints (Python) or JSDoc (JavaScript)

6. ERROR HANDLING
   - Bare except/catch clauses swallowing all errors silently
   - Missing error handling on I/O operations, network calls, DB queries
   - Returning None/null to signal errors instead of raising exceptions
   - Overly generic error messages with no useful context
   - Missing finally blocks for cleanup

7. CODE STRUCTURE & ORGANISATION
   - Long files that should be split into modules
   - Misplaced responsibilities (UI logic in data layer, etc.)
   - Missing abstractions (raw dicts/lists where a dataclass/model would clarify intent)
   - Inappropriate use of global variables
   - Hard-coded configuration that should be in a config file or env variable

8. TESTABILITY
   - Functions that are hard to unit test due to tight coupling
   - Missing separation between pure logic and side effects
   - No clear seams for dependency injection
   - Functions doing too much to mock effectively

9. PYTHON-SPECIFIC (if Python)
   - Not using list/dict/set comprehensions where appropriate
   - Using mutable default arguments
   - Not using context managers (with statement) for resources
   - Missing dataclasses or NamedTuples where plain dicts are used
   - Missing __str__, __repr__ on custom classes

10. JAVASCRIPT/TYPESCRIPT-SPECIFIC (if JS/TS)
    - Callback hell instead of async/await
    - Missing TypeScript types or using `any` extensively
    - console.log left in production code
    - Not using optional chaining (?.) and nullish coalescing (??)
    - var instead of const/let
