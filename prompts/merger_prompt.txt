YOUR TASK
═══════════════════════════════════════════════

Step 1 — Conflict resolution
Before writing any code, identify if any suggestions from two different reviews CONFLICT with each other (e.g. a performance fix that hurts readability, or a security fix that changes the function signature in a way that affects the style review). For each conflict, choose the safer or more impactful option and note your decision briefly.

Step 2 — Produce the refactored code
Write the complete, improved version of the code. Rules:
- Apply ALL valid findings from all three reviews in one unified pass.
- Do NOT partially fix things — if a function has a security issue AND a naming issue, fix both.
- Maintain the original code's intent and public API unless a security issue specifically requires a change.
- Add or update comments/docstrings where the style review flagged them as missing.
- Do not add new features or logic that was not in the original code.
- The refactored code must be complete and runnable — not a diff, not a snippet.

Step 3 — Annotate with inline comments
For every significant change you make, add a short inline comment explaining WHICH review flagged it and WHY it was changed. Use this format:
  # [SECURITY] Parameterized query prevents SQL injection
  # [PERFORMANCE] dict lookup O(1) replaces list search O(n)
  # [STYLE] Renamed from 'x' to 'user_id' for clarity
  # [SECURITY + STYLE] Removed hardcoded secret, moved to env var

═══════════════════════════════════════════════
RULES
═══════════════════════════════════════════════
- The refactored code in Section 3 must be 100% complete — not truncated, not abbreviated with "# ... rest of code unchanged". Write every line.
- Never remove functionality that existed in the original code unless a security review explicitly required it.
- If a review flagged an issue as LOW severity and the fix would significantly complicate the code, you may skip it — but note it in Remaining Risks.
- Maintain the original language, framework, and dependencies unless a review specifically recommended changing them.
- The inline [TAG] comments are mandatory — they are how the developer learns what changed and why.
- Be the voice of reason: if a review suggested something impractical or over-engineered for the context, you may omit it with a note.
